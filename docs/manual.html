项目概览

TG Manager 是面向个人用户和小团队运营者的 Telegram 管理脚本。它由两个 Telethon 客户端驱动：Userbot（个人账号 MTProto）负责高权限操作，Bot（Bot Token）负责用户交互。所有功能以插件形式存在，通过事件总线解耦通信。

安装部署

环境要求

  • Python 3.11 或更高版本
  • Telegram api_id 和 api_hash（从 my.telegram.org 获取）
  • 一个 Bot Token（从 @BotFather 创建）
  • 一个可登录的 Telegram 用户账号（用于 Userbot）

安装步骤

terminal
# 1. 克隆仓库

# 2. 安装依赖（推荐 editable 模式）

双客户端架构

核心原则：Userbot 负责执行，Bot 负责交互。能用 Bot 做的绝不用 Userbot。

能力Userbot (MTProto)Bot
身份个人账号@YourManagerBot
消息监听所有群/频道（含受限）被添加为管理员的群
受限内容可读取不可
交互界面无InlineKeyboard / Command
封号风险高（需限速）极低
典型操作转发 / 镜像 / 监控 / 管理命令响应 / 通知推送 / 菜单

DualClient 消息发送策略

send_message(chat_id, text, prefer_bot=True)
  │
  ├─ prefer_bot=True  → 先 Bot，失败后 fallback Userbot
  └─ prefer_bot=False → 先 Userbot，失败后 fallback Bot

notify_admin(text)  → 永远使用 Bot（官方通知通道）

FloodWait 处理链

Telegram API 返回 FloodWaitError(seconds=N)
  ↓
客户端捕获 → 发射 "flood_wait" 事件
  ↓
RateLimiter.handle_flood_wait(N):
  · 实际等待 = N × 1.5 + 随机抖动
  · 连续触发计数 +1
  · 连续 ≥ 3 次 → 全局暂停 5 分钟
  ↓
成功发送后 → reset_flood_counter()

速率限制

防封号第一：所有 Telegram API 调用必须经过 RateLimiter。采用多维限速 + FloodWait 自适应退避 + TOCTOU 安全的时间槽预留机制。

限制维度默认值说明
全局消息30 条/分钟滑动窗口，60 秒内最多 30 条
单聊天间隔3.0 秒同一聊天两次发送最小间隔
加入群组20 个/天每日加群上限，超限抛 RateLimitError
添加成员50 人/天每日加人上限，间隔 30 秒
文件下载并发 3 个Semaphore 控制并发数
FloodWait 倍数1.5×Telegram 要求等 N 秒，实际等 N×1.5 秒
FloodWait 熔断连续 3 次连续触发 3 次 → 全局暂停 5 分钟
随机抖动0.5 ~ 2.0 秒所有间隔附加随机延迟
🔒
TOCTOU 防护：时间戳在锁内预留（_global_timestamps.append(future_ts)），确保多个并发协程不会同时通过限速检查。实际 sleep 在锁外执行，避免序列化。

消息管理

message.sender — 消息发送

订阅 send_message 事件，通过 DualClient 发送消息。支持参数：

参数类型必填说明
chat_idint是目标聊天 ID
textstr是消息内容
prefer_botbool否优先使用 Bot（默认 True）
file—否附件媒体
reply_toint否回复的消息 ID

message.forwarder — 消息转发

基于数据库 ForwardRule 自动转发非频道消息（频道由 mirror 处理）。支持三种转发模式和两种过滤类型：

forward

标准转发，保留「转发自 XXX」标签

copy

复制消息，无转发标签，保留媒体和格式

copy_clean

委托 CopyCleanPlugin 去除所有来源标记

过滤类型

类型匹配方式示例
keyword大小写不敏感子串pattern="重要" → 匹配含"重要"的消息
regexPython 正则搜索pattern="价格\d+" → 匹配"价格100"
none不过滤，转发所有—

message.recorder — 消息入库

监听所有消息（收发），持久化到数据库供搜索和 AI 总结复用。记录字段：

message_id, chat_id, sender_id, date, text, media_type, is_forward, forward_from_chat_id, reply_to_msg_id, grouped_id, views, is_pinned

📢

频道管理

channel.mirror — 频道镜像

监控源频道新消息，自动同步到目标频道。仅处理频道/超级群消息（event.is_channel），忽略自身发送。

源频道消息 → 查询 ForwardRule → 按 forward_type 执行
  ├─ forward → userbot.forward_message()
  ├─ copy    → userbot.send_message() + media + entities
  └─ copy_clean → emit("copy_clean_message") → CopyCleanPlugin 处理

channel.copy_clean — 去标签复制

订阅 copy_clean_message 事件，获取原消息后重新发送（无转发标签），保留文本、媒体和格式实体。

自动回复

rule_engine — 规则匹配

监听消息，根据 AutoReplyRule 数据库规则自动匹配并触发回复。支持三种匹配方式：

rule_type匹配逻辑示例
keyword大小写不敏感子串搜索"你好" → 匹配所有含"你好"的消息
regexPython 正则表达式"价格\d+元" → 匹配"价格100元"
exact完全匹配（忽略大小写）"帮助" → 仅匹配单独的"帮助"

规则按 priority 排序，高优先级先匹配。匹配成功后发射 auto_reply_matched 事件。

template_reply — 模板回复

订阅 auto_reply_matched 事件，用规则中定义的 response 模板生成回复。用户输入中的 {} 字符会被转义，防止模板注入。

⏰

定时任务

基于 APScheduler 3.x 的 AsyncIOScheduler，Cron 任务持久化到数据库，崩溃重启后自动恢复。

Cron 表达式格式

┌───── 分 (0-59)
│ ┌───── 时 (0-23)
│ │ ┌───── 日 (1-31)
│ │ │ ┌───── 月 (1-12)
│ │ │ │ ┌───── 周 (0-6, 0=周一)
│ │ │ │ │
* * * * *

示例:
0 9 * * *     # 每天 9:00
*/30 * * * *  # 每 30 分钟
0 8 * * 1-5   # 工作日 8:00

管理命令

/schedule list列出所有启用的定时任务
/schedule add <名称> <分> <时> <日> <月> <周> <消息>创建定时任务
/schedule remove <ID>停止并删除指定任务
💡
崩溃安全：添加时先验证 cron 再写 DB；删除时先标记 DB 禁用再从调度器移除。确保 DB 与调度器状态一致。

AI 总结

chat_summarizer — 群聊总结

通过 /summarize [N] [chat_id] 命令触发。先从数据库读取最近 N 条消息（由 recorder 插件入库），如不足则回退实时拉取。使用 LLM 生成中文摘要。

参数顺序无关：负数（如 -1001234567890）解析为目标 chat_id，正整数（如 100）解析为消息条数上限。缺省时使用当前聊天和默认 200 条。

# 当前聊天，200 条（默认）
/summarize

# 当前聊天，100 条
/summarize 100

# 远程指定群组，200 条 — Bot 不在群里也能用
/summarize -1001234567890

# 远程群组 + 自定义条数（顺序无关）
/summarize 100 -1001234567890
💡
远程总结依赖 Userbot 已加入目标群。Bot 即使被踢出群组，只要 Userbot 仍是成员，recorder 持续入库 + chat_summarizer 远程拉取就能继续工作。聊天标题由 Userbot 通过 get_entity 自动解析。

content_summarizer — URL 总结

通过 /url <链接> 命令触发。流程：

1. SSRF 防护：验证 URL 非内网地址（is_safe_url）
2. 安全重定向：每跳验证目标 IP（最多 5 跳）
3. 内容提取：readability → BeautifulSoup → 纯文本
4. 截断至 8000 字符
5. LLM 生成摘要

ask_handler — AI 问答

通过 /ask <问题> 命令触发。直接将问题发送给 LLM，返回简洁回答。

监控告警

实时监控消息，关键词命中后通过 Bot 把告警推送给管理员。关键词存储于数据库 monitor_keywords 表，运行时通过 /keyword 命令在 Telegram 中直接管理，无需修改 config.yaml 或重启服务。

命令式管理（推荐）

# 列出全部关键词
/keyword
/keyword list

# 添加关键词（保存到 DB，立即生效）
/keyword add 紧急
/keyword add 故障

# 移除关键词
/keyword remove 紧急
/keyword del 紧急   # 别名
/keyword rm 紧急    # 别名

命令处理流程：KeywordHandler 收到 /keyword → 通过事件总线发出 keyword_add/remove/list → KeywordAlertPlugin 写入数据库并刷新内存中的关键词列表 → 通过 Bot 回复操作结果。

正则模式（高级，仍走配置）

复杂正则匹配仍由 config.yaml 管理（修改后需重启）：

plugin_config:
  monitor:
    regex_patterns:
      - "价格[超过]*\d+万"
      - "error\s*code\s*\d+"
    monitored_chats: []  # 空 = 监控所有聊天
💡

安全机制

SSRF 防护

is_safe_url() 在 URL 抓取前解析 DNS 并验证目标 IP 非私网/回环/链路本地/保留地址。重定向跟随时每跳重新验证。

模板注入防护

欢迎消息和自动回复中，用户控制的字段（用户名、群名）在传入 str.format() 前转义 {} → {{}}。

权限检查

admin_only 装饰器和 _check_admin 方法确保管理员命令仅 admin_user_id 可执行。

命令节流

throttle 装饰器限制命令调用频率，防止滥用。

凭据隔离

敏感信息（API Key、Token、手机号）存放 .env，.session 文件在 .gitignore 中排除。

异常泄露防护

/reload 失败时仅输出通用错误消息，不暴露原始异常栈。回调数据解码使用 errors="replace"。

常见问题

启动时报 "配置字段 api_id 必须为整数"？

.env 中的 TG_API_ID 必须是纯数字。确认没有引号包裹，且 .env 文件已正确加载（与项目根目录同级）。

Bot 不响应命令？

1. 确认 Bot Token 正确且 Bot 已启动
2. 检查你的 Telegram 账号 ID 是否与 TG_ADMIN_USER_ID 一致（管理员命令仅限此 ID）
3. 检查日志中是否有 registered command 信息

FloodWait 导致功能暂停？

这是 Telegram 的频率限制。系统会自动等待 N × 1.5 秒后重试。连续触发 3 次会全局暂停 5 分钟。解决方法：降低操作频率，或调整 rate_limit 配置。

如何切换 LLM 模型？

修改 config.yaml 中 llm.default_provider 的值。可选：openai、claude、gemini、deepseek、ollama。确保对应的 API Key 已配置。

如何添加转发规则？

当前版本通过直接在数据库 forward_rules 表中插入记录来管理。需要设置 source_chat_id、target_chat_id、forward_type。后续版本将添加 Bot 命令管理界面。
